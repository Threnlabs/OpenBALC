# OpenBALC — Hybrid RAG Ingestion Pipeline

This document details the architecture, hybrid chunking strategy, ingestion management UI, and tracking configuration of OpenBALC's RAG Ingestion Engine.

---

## 1. High-Level Pipeline Flow

OpenBALC employs a non-blocking, multi-stage ingestion system to transition unstructured source materials (PDFs, URLs, raw text) into enriched, indexable vector chunks in the PostgreSQL database.

```
[ Raw Source (PDF/URL/Text) ] 
          │
          ▼
┌──────────────────────────────┐
│   Stage A: Text Extraction   │ ──► Parse text & Markdown structures
└──────────────────────────────┘
          │
          ▼
┌──────────────────────────────┐
│  Stage B: Asset Description  │ ──► Describe images/tables via Gemini Flash
└──────────────────────────────┘
          │
          ▼
┌──────────────────────────────┐
│  Stage C: Hybrid Chunking    │ ──► Segment by chunk types (Tables/Images/Content)
└──────────────────────────────┘
          │
          ▼
┌──────────────────────────────┐
│     Stage D: Embedding       │ ──► text-embedding-004 (768 dimensions)
└──────────────────────────────┘
          │
          ▼
┌──────────────────────────────┐
│    Stage E: Database Write   │ ──► Batch insert chunks & update telemetry
└──────────────────────────────┘
```

## 2. The Hybrid Chunking Strategy

Unlike basic character-based chunking that breaks document structure, OpenBALC uses a **type-aware hybrid chunking** approach to maintain high semantic coherence:

### A. Content Chunks (`chunk_type = 'content'`)
* **Strategy:** Recursive splitting based on paragraph and line boundaries.
* **Token Budget:** Configured in the Admin portal (Default: `500` tokens, with `100` tokens overlap).
* **Overlap:** A sliding overlap window maintains contextual continuity between sequential chunks.

### B. Table Chunks (`chunk_type = 'table'`)
* **Strategy:** Tables are treated as **indivisible atomic units**.
* **Chunk Bounds:** Regardless of physical character or token length, the entire markdown table is kept in a single chunk.
* **Asset Enrichment:** A 2-to-3 sentence summary explaining relationships and key data (generated by Gemini Flash) is appended to the table markdown as a suffix block (`> **📊 Table Summary**: ...`) to boost vector search retrieval capability.

### C. Image Caption Chunks (`chunk_type = 'image_caption'`)
* **Strategy:** Extracted images are uploaded to public Supabase Storage buckets, and their captions are indexed as dedicated chunks.
* **Data Fields:** Contains the caption text, contextual header information, and a direct `asset_url` pointing to the public storage location.
* **LLM Hydration:** When retrieving an image caption chunk during chat, the `asset_url` is passed in the context, allowing the assistant to render the actual figure directly in the chat panel.

## 3. Database Schema

The hybrid chunks reside in the `module_chunks` table, which is mapped with vector similarity metrics:

```sql
CREATE TABLE module_chunks (
  id              SERIAL PRIMARY KEY,
  module_id       INTEGER REFERENCES modules(id) ON DELETE CASCADE,
  source_id       INTEGER REFERENCES module_sources(id) ON DELETE CASCADE,
  content         TEXT NOT NULL,
  token_count     INTEGER NOT NULL,
  chunk_type      VARCHAR(20) NOT NULL DEFAULT 'content' CHECK (chunk_type IN ('content', 'table', 'image_caption')),
  chunk_index     INTEGER NOT NULL DEFAULT 0,
  topic_id        INTEGER REFERENCES topics(id) ON DELETE SET NULL,
  asset_url       TEXT,
  times_retrieved INTEGER NOT NULL DEFAULT 0,
  embedding       vector(768),
  created_at      TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP NOT NULL
);
```

### Telemetry & Color-Mapping Heatmaps
* **`times_retrieved`**: Each time a chunk is utilized in a successful RAG chat generation, its `times_retrieved` count is incremented via an RPC database call (`increment_chunk_retrieved`).
* **Visual Color Mapping:** The front-end queries retrieval stats aggregated per `topic_id` using the `useGetChunkHeatmap` React Hook. This heat map exposes which chapters or concepts are queried most frequently, displaying color codes (e.g. green to red) to students and educators.

## 4. Admin Management Controls

The Admin Portal (`/admin` tab -> **Modules & Ingestion Control**) provides explicit, real-time buttons and config panels:

* **Hybrid Chunking Settings:** Interactive sliders to adjust *Target Chunk Size* and *Overlap* (persisted dynamically in the database via the `ingestion_config` table).
* **Ingest / Re-Ingest (Per Source):** Triggers extraction, asset captioning, and embedding pipelines for individual files.
* **Remove (Per Source):** Deletes all embeddings for the source from `module_chunks` and resets its ingestion status back to `pending`.
* **Ingest All (Per Module):** Ingests all pending source documents in batch sequence.
* **De-ingest All (Per Module):** Deletes all chunks associated with the module, resetting it back to a `draft` status.
